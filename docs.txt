ClarityReply Project Notes
==========================

This document summarizes the ClarityReply codebase from the actual project
files. It is written so a new reader can understand what the project is, how it
is structured, and which files matter most.


1. Project Summary
------------------

Project name:
ClarityReply

Builder:
Tejasvi Bajaj

Competition:
Anakin Blitz

Format:
6-hour hackathon

Main idea:
ClarityReply is a Django web app that extracts social post context from a URL
or screenshot and generates three platform-aware reply variations.

The app supports:
- URL extraction
- Screenshot OCR extraction
- Platform-aware reply generation
- Discord and WhatsApp conversation context
- Instagram-focused reply behavior
- Django admin inspection of requests and generated replies

The generated reply API returns:

{
  "variation_1": "",
  "variation_2": "",
  "variation_3": ""
}


2. High-Level Flow
------------------

User opens the frontend page.

User chooses either:
- URL
- Screenshot

For URL:
- Frontend sends POST /api/extract/
- Backend validates URL
- Backend detects platform from URL
- Anakin URL Scraper fetches cleanedHtml
- Platform parser extracts title, content, author, summary, and engagement
- Data is saved to ReplyRequest
- Data is returned to frontend

For screenshot:
- Frontend sends image to POST /api/extract/
- Backend stores it temporarily
- Gemini OCR extracts visible text
- OCR text is saved to ReplyRequest
- Data is returned to frontend

For reply generation:
- Frontend sends POST /api/generate-reply/
- Backend updates ReplyRequest with editable fields and preferences
- Discord/WhatsApp may include previous_messages
- ClarityReplyService asks Gemini for three variations
- Replies are saved into existing model fields
- API returns variation_1, variation_2, variation_3


3. Supported Platforms
----------------------

The active platform list in backend validation and frontend UI is:

- linkedin
- twitter
- instagram
- youtube
- reddit
- facebook
- discord
- whatsapp
- tiktok
- custom

ProductHunt and HackerNews are not part of the current supported platform list.

8. Views and API Behavior
-------------------------

core/views.py
-------------

Main Django views file.

Important constants:
- PLATFORM_TEMPLATES contains the active platform list.
- ALLOWED_IMAGE_TYPES allows jpeg, png, webp, and gif.
- URL_VALIDATOR validates URL input.

Important helpers:
- success_response returns JSON with success: true.
- error_response returns JSON with success: false and error text.
- parse_request_data supports JSON, multipart form data, and form-urlencoded.
- validate_url checks URL validity.
- build_platform_data returns extracted platform data for frontend rendering.
- debug_reply_request_values creates logging dictionaries for extraction.

HomeView:
- Renders templates/core/index.html.

TemplateListView:
- Returns supported platform templates.

ExtractContentView:
- Accepts POST requests.
- Requires source_type to be url or screenshot.
- For URLs:
  - Validates url.
  - Detects platform from URL.
  - Creates ReplyRequest.
  - Calls ContentExtractionService.extract_and_save.
- For screenshots:
  - Requires image file.
  - Validates image content type.
  - Stores image temporarily.
  - Calls OCR extractor.
  - Removes temporary file in finally block.
- Applies platform preset.
- Saves and re-queries the request.
- Returns request_id and platform_data.
- Logs extraction steps.

GenerateReplyView:
- Accepts POST requests.
- Uses existing ReplyRequest if request_id is provided.
- Otherwise can create a text-based ReplyRequest from post_content.
- Copies editable frontend fields onto the model.
- If previous_messages is provided for discord or whatsapp:
  - Stores it in conversation_summary.
  - Stores latest content in latest_message.
  - Stores a list item in conversation_history.
- Validates selected_template against PLATFORM_TEMPLATES.
- Applies reply settings.
- Applies platform preset.
- Saves request.
- Calls ClarityReplyService.generate_and_save.
- Returns request_id, variation_1, variation_2, variation_3.


9. Extraction Service
---------------------

core/services/extract_description.py
------------------------------------

Main extraction service.

Important classes and functions:

OCRExtractor:
- Uses Gemini vision to extract text from screenshots.
- Supports local file paths, URLs, and data URI/base64 input.
- Saves OCR text into req.ocr_text.
- Copies OCR text into req.content if content is empty.

_parse_count:
- Converts strings like 1.2K or 12,345 into integers.

AnakinScrapeJob:
- Dataclass representing the Anakin URL Scraper response.
- Important field is cleaned_html.

ScrapedContent:
- Dataclass for normalized extraction output.
- Stores title, content, summary, image_url, author fields, engagement,
  views, job metadata, and raw response.

detect_platform:
- Detects platform from URL host.
- Supports linkedin, twitter/x, instagram, youtube/youtu.be, reddit, tiktok,
  and facebook.
- Returns generic for unknown hosts.

AnakinScraperClient:
- Calls Anakin URL Scraper.
- Sends url and useBrowser.
- Supports immediate completed response.
- Supports polling if Anakin returns a job ID.
- Raises errors for failed jobs, missing job IDs, and timeout.

URLContentExtractor:
- Calls Anakin and then platform-specific parsers.
- Uses cleanedHtml as the source of truth.
- Does not use an external fallback scraper.

Platform parsers:
- _parse_twitter: looks for article markup, author links, tweet body, media,
  reply/like/repost counts, and views.
- _parse_linkedin: looks for h1, LinkedIn commentary blocks, author name,
  reaction count, and image.
- _parse_youtube: reads VideoObject JSON-LD first, then visible title,
  description, and owner fallbacks.
- _parse_instagram: reads Instagram embedded application/json SSR payloads,
  extracts caption.text, author username/name, like_count, comment_count,
  and display_uri.
- _parse_generic: reads JSON-LD Article/NewsArticle/BlogPosting data when
  present, otherwise uses visible text.

ContentExtractionService:
- Routes extraction by req.source_type.
- For screenshot, calls OCRExtractor.
- For url, calls URLContentExtractor and saves title, content, summary,
  author fields, likes, comments, and shares.
- Re-queries the saved ReplyRequest before returning it.


10. Reply Generation Service
----------------------------

core/services/reply.py
----------------------

Main reply generation service.

Important constants:
- PLATFORM_RULES defines platform-specific reply guidance.
- Instagram rule says replies should be short, human, conversational,
  emoji-natural, and not corporate or LinkedIn-like.

Helper functions:
- get_platform_rule normalizes platform and returns a platform rule.
- _emoji_level_label converts emoji_level integer into text guidance.
- _creativity_level_label converts creativity_level integer into text guidance.
- _length_label converts reply_length into sentence guidance.

ClarityReplyService:
- Uses Gemini through google-genai.
- Builds a focused prompt.
- System prompt says only generate replies, no analysis or meta commentary.
- User prompt asks for three alternative phrasings of the same reply strategy.
- Requires JSON keys:
  - variation_1
  - variation_2
  - variation_3
- For Discord and WhatsApp, _combined_context combines:
  - conversation_summary
  - latest_message
  - content
- generate calls Gemini with response_mime_type application/json.
- _extract_json parses model output and can recover JSON inside extra text.
- _validate requires the three variation keys.
- generate_and_save stores:
  - variation_1 in professional_reply
  - variation_2 in friendly_reply
  - variation_3 in engaging_reply


11. Platform Presets
--------------------

core/common/platform_presets.py
-------------------------------

Defines default reply settings per platform.

Important pieces:
- PlatformPreset dataclass stores audience_type, formality_level, reply_goal,
  reply_style, reply_length, emoji_level, creativity_level, and notes.
- PLATFORM_PRESETS includes linkedin, twitter, reddit, discord, whatsapp,
  youtube, instagram, facebook, and tiktok.
- x is an alias for twitter.
- DEFAULT_PRESET is used when platform is unknown.
- get_preset normalizes platform names and returns the matching preset.
- apply_platform_preset fills empty/default fields on ReplyRequest.

Important platform examples:
- LinkedIn: formal, professional, medium length, low emoji.
- Twitter/X: casual, short, witty, higher creativity.
- Reddit: casual, informative, no emoji.
- Discord: casual, short, chat-like.
- WhatsApp: personal, warm, short.
- YouTube: supportive, short.
- Instagram: short, friendly, emoji-forward.
- Facebook: warm and straightforward.
- TikTok: playful, short, high creativity.

15. Current Product Behavior
----------------------------

Current user-facing flow:

1. User chooses URL or Screenshot.
2. User extracts content.
3. App shows extracted settings.
4. If platform is Discord or WhatsApp, Platform Data Editor is hidden.
5. Discord and WhatsApp show Conversation Context.
6. User configures platform/style/length/emoji/creativity.
7. User generates replies.
8. App shows three variation cards.
9. Copy button copies text and opens original URL if one exists.


16. Important Implementation Notes
----------------------------------

- The app does not use Django REST Framework.
- Views return JsonResponse directly.
- CSRF is exempted on the class-based views.
- Anakin cleanedHtml is treated as the source of truth for URL extraction.
- Screenshot extraction uses Gemini OCR.
- Reply generation uses Gemini and expects valid JSON.
- The model still has professional_reply, friendly_reply, and engaging_reply,
  but the app now treats them as variation storage.
- Manual input has been removed from the frontend.
- ProductHunt and HackerNews are not supported in the current platform lists.
- No extra database migrations were added after the initial ReplyRequest model.
